# 自転車ナビゲーションシステム 開発指示書（UI 改善フェーズ）

日本の交通法規に準拠した自転車ナビゲーションシステム。
卒業研究（青山学院大学・宮治研究室）のシステム実装プロジェクト。

- タスク C1（way_id 解決の二分探索化）完了
- タスク C2（interval[0] 妥当性確認、approach → departure road へのバグ修正兼ねる）完了

## このフェーズの方針

### 目的

1. **スマホでの実用利用に耐える UI** に作り変える
2. **既存ナビ（Google Maps）と同等の使い勝手** を確保し、論文での「実用性評価」軸を強化する
3. 提案書フェーズ2で予定している **SUS アンケート** にかけられる完成度に引き上げる

### 設計方針

- **地図ファースト**：画面全体を地図にし、その上にボトムシート・FAB・トップバー
  などのオーバーレイ要素を配置する
- **Google Maps スタイルのボトムシート**：下からせり上がる方式で、3段階の高さ
  （peek / half / full）を持つ
- **preparing モード**：ボトムシートに入力フォーム・違反箇所一覧・距離/所要時間を
  まとめる。地図にはルートと違反マーカーを常時表示
- **riding モード**：ボトムシート最小化、地図を最大化。違反箇所までの距離と音声
  アラートで運転中の視認性を最優先する

### 不変の制約（変更禁止）

- バックエンド API の仕様は変更しない（`POST /api/route` のリクエスト・レスポンス
  形式は維持）
- v3 判定ロジック・law_checker・rerouter は変更しない
- モード切替の状態管理（`mode` state）の概念は維持（preparing / riding）
- `check_sidewalk_violation` は引き続き呼び出さない
- experiment.py / batch エンドポイントは維持

## タスク一覧

### タスク U1：レイアウト基盤の刷新（地図ファースト + ボトムシート）【優先度：最高】

#### 背景

現状の `App.jsx` は地図・コントロール・違反アラートが個別パネルで配置されており、
スマホで使うと地図が小さく、情報も散らばっている。Google Maps のように
「地図が画面全体、その上にオーバーレイ」のレイアウトに刷新する。

#### 変更対象ファイル

- `frontend/src/App.jsx`（レイアウト全面書き直し）
- `frontend/src/components/BottomSheet.jsx`（新規作成）
- `frontend/src/components/MapView.jsx`（フルスクリーン対応、コントロール要素の
  オーバーレイ化）
- `frontend/src/components/RidingView.jsx`（同上、riding モードでもボトムシートを
  使う方針なので統合検討）
- `frontend/src/index.css` または `App.css`（グローバルスタイル調整）

#### 手順

1. **App.jsx のルートレイアウトを刷新**：
   - 画面全体（100vw × 100vh）を地図エリアに変更
   - 地図の上に以下をオーバーレイ配置：
     - 上部：トップバー（モード表示・設定ボタンなど。最初は最小限）
     - 下部：ボトムシート
     - 右下：FAB（現在地ボタンなど）
2. **BottomSheet.jsx を新規作成**：
   - 下からせり上がる UI コンポーネント
   - 3段階の高さスナップ：
     - peek（約 15% 高さ、ハンドルと最小限の情報のみ）
     - half（約 50% 高さ、入力フォームと違反一覧の概要）
     - full（約 90% 高さ、全情報を表示）
   - ドラッグで高さ変更可能、タップでも段階切替可能
   - ハンドル（横長のバー）を上部に配置
   - React の useState / useRef で高さ管理、CSS transform で滑らかにアニメーション
   - **localStorage は使わない**（卒研の動作環境でブラウザ依存を避けるため、
     セッション中のメモリ管理のみ）
3. **MapView.jsx をフルスクリーン化**：
   - 親要素の 100% に追従するレイアウト
   - 既存のコントロール（出発地・目的地ピン、ルート線、違反マーカー）は維持
4. **既存の入力フォーム・違反パネルを BottomSheet 内に移植**：
   - 現状の preparing モード用パネルの中身をボトムシートに移動
   - レイアウトはタスク U2 で詳細調整
5. **タッチ操作の最適化**：
   - 地図のピンチ・パン操作とボトムシートのドラッグ操作が競合しないこと
   - ボトムシートのハンドル領域のみがドラッグ対象になる

#### 完了条件

- スマホ画面（375px 幅）で地図が画面全体に表示される
- ボトムシートが下からせり上がり、ドラッグで peek / half / full の3段階に切り替わる
- 既存の入力フォーム・違反パネルの機能がボトムシート内で同等に動作する
- preparing モードの動作シナリオ（出発地・目的地入力 → ルート探索 → 違反確認）が
  すべて維持されている
- デスクトップブラウザ（1280px 幅）でも崩れない

### タスク U2：preparing モードの UI 構築【優先度：高】

#### 背景

ボトムシート方式で、地図には常にルートと違反マーカーが見えていてほしい。
ボトムシートの中には以下の情報をまとめる。

#### 表示要素（タスク U1 のボトムシート内に配置）

##### 地図に常時表示する要素

- ルートの全体プレビュー（線で表示）
- 違反箇所のマーカー（confidence ≥ 0.7 は赤、< 0.7 は黄色のような視覚的差異）
- 出発地・目的地ピン
- 現在地ピン（GPS 取得時）

##### ボトムシート内の要素

- **peek 段階（最下層）**：
  - 「ルートを探索」ボタン（ルート未取得時）
  - または「○○ km / 約○○分」のサマリー（ルート取得後）
  - 現在地ボタンへのショートカット
- **half 段階（中間）**：
  - 出発地・目的地入力フォーム（折りたたみ可）
  - 距離・所要時間サマリー
  - 違反箇所の件数サマリー（「⚠️ 違反候補 2件」など）
- **full 段階（最上層）**：
  - 上記すべて
  - 違反箇所の詳細リスト（タップで地図上の該当マーカーにフォーカス）
  - 各違反箇所の説明パネル（タップで展開、ルール種別・信頼度・代替経路の説明）

##### 地図右下の FAB

- 現在地ボタン（地図を現在地にセンタリング）
- ルート探索ボタン（出発地・目的地が両方セットされている時のみ表示）

#### 変更対象ファイル

- `frontend/src/components/BottomSheet.jsx`（U1 で作成済み）
- `frontend/src/components/PreparingPanel.jsx`（新規作成 or 既存パネルから移行）
- `frontend/src/components/ViolationCard.jsx`(新規作成、違反1件ぶんの説明カード)
- `frontend/src/components/RouteSummary.jsx`（新規作成、距離・時間表示）
- `frontend/src/components/MapView.jsx`（マーカーの色分け対応）

#### 手順

1. **PreparingPanel.jsx を新規作成**：
   - ボトムシートの中身を担当
   - 高さの段階に応じて表示要素を切り替える（peek / half / full で異なる情報を出す）
2. **ViolationCard.jsx を新規作成**：
   - 1件の違反を表すカード UI
   - 表示項目：違反種別（一方通行 / 二段階右折）、信頼度、地点の住所もしくは座標、
     簡潔な説明
   - タップすると地図上の該当マーカーにフォーカスする（zoom + pan）
3. **RouteSummary.jsx を新規作成**：
   - 「○○ km / 約○○分」を大きく表示
   - peek 段階で常に見える位置に配置
4. **MapView.jsx のマーカー色分け**：
   - `confidence ≥ 0.7` → 赤
   - `confidence < 0.7` → 黄色
   - マーカータップでボトムシートが該当 ViolationCard まで自動スクロール
5. **FAB の配置**：
   - 地図右下に現在地ボタン
   - 出発地・目的地が両方セットされたらルート探索 FAB を表示

#### 完了条件

- 地図全体にルートと違反マーカーが常時見える
- ボトムシートが peek / half / full で適切に情報量を切り替える
- 違反カードをタップすると地図がフォーカスする
- 地図上のマーカーをタップするとボトムシートが該当カードを表示する
- 現在地ボタンが動作する
- スマホ実機（または DevTools のレスポンシブモード）で操作しやすい

### タスク U3：riding モードの実用化【優先度：高】

#### 背景

走行中は画面を見ない前提で、視認性最優先。情報量を絞り、音声・振動で
法規違反が近づいたことを伝える設計にする。

#### 表示要素

##### 優先度1：大きく見やすい地図

- ボトムシートを peek（最小）に固定。地図を最大化
- 地図中心は現在地に追従、進行方向に応じて自動回転（heading-up 表示）
- 現在地マーカーは大きな矢印型（進行方向が一目でわかる）
- ルート線は太め、違反マーカーは目立つアイコン

##### 優先度2：次の違反箇所までの距離表示

- 画面上部または地図上に「次の違反箇所まで ○○ m」を大きく表示
- 100m 以内に近づくと表示を強調（色変化や点滅）
- 違反種別アイコンも同時に表示（一方通行マーク / 二段階右折マーク）

##### 優先度3：音声/振動アラート

- 違反箇所の 200m 手前で音声アラート
  - 例：「150m 先、一方通行に注意してください」
  - Web Speech API（SpeechSynthesisUtterance）を使用
- 違反箇所の 50m 手前で振動アラート
  - Navigator.vibrate API を使用
  - スマホでのみ動作（デスクトップでは何もしない）
- 音声/振動はトグルでオン/オフ切替可能

##### 優先度4：現在の進行方向・速度表示

- 地図右下に小さく速度表示（GPS から算出、km/h）
- 進行方向は地図の回転で表現するため、テキスト表示は最小限でよい

#### 変更対象ファイル

- `frontend/src/components/RidingView.jsx`（フルスクリーン化と音声/振動対応）
- `frontend/src/components/NextViolationIndicator.jsx`（新規作成、次の違反までの
  距離表示）
- `frontend/src/components/SpeedIndicator.jsx`（新規作成、速度表示）
- `frontend/src/services/voiceAlert.js`（新規作成、音声アラート機能）
- `frontend/src/services/vibrateAlert.js`（新規作成、振動アラート機能）
- `frontend/src/services/geolocation.js`（既存があれば、heading の追従を強化）

#### 手順

1. **RidingView.jsx をフルスクリーン化**：
   - ボトムシートを peek 状態に固定
   - 地図を画面全体に表示
   - 地図の回転を有効化（Leaflet の場合は `leaflet-rotate` または手動で
     transform を適用）
2. **現在地マーカーを矢印型に**：
   - 進行方向に応じて回転する SVG / Canvas マーカー
   - GPS の heading（方角）データを使用
3. **NextViolationIndicator.jsx を新規作成**：
   - 現在地から次の違反箇所までの距離をリアルタイム計算
   - Haversine 公式またはルート上の距離を使用
   - 200m / 100m / 50m の閾値で表示を変化させる
4. **音声アラート（voiceAlert.js）を実装**：
   - `SpeechSynthesisUtterance` で日本語の音声を生成
   - 同じ違反箇所に対して複数回鳴らさないように、アラート済みフラグを管理
   - 200m 手前で1回、50m 手前で1回が目安
5. **振動アラート（vibrateAlert.js）を実装**：
   - `navigator.vibrate([100, 50, 100])` のようなパターンで振動
   - HTTPS 環境かつスマホでのみ動作する
6. **SpeedIndicator.jsx を新規作成**：
   - GPS の `speed` プロパティを表示(m/s → km/h 変換)
   - 表示は地図右下、小さめ
7. **音声/振動のオン/オフトグル**：
   - 地図左上または上部バーに設定ボタン
   - トグル状態は React state で管理（localStorage は使わない）

#### 完了条件

- riding モード起動時に地図がフルスクリーンになる
- 現在地マーカーが進行方向に回転し、地図も heading-up で表示される
- 次の違反箇所までの距離がリアルタイムで表示される
- 200m / 50m の閾値で音声・振動アラートが鳴る
- 音声/振動のオン・オフを切り替えられる
- スマホ実機での走行テスト（または徒歩テスト）で実用に耐える

### タスク U4：スマホ実機テスト・チューニング【優先度：中】

#### 背景

U1〜U3 の実装が完了したら、必ずスマホ実機で動作確認する。デスクトップの
DevTools だけでは検出できない問題（タッチ感度・GPS 精度・電池消費）が
存在するため。

#### 手順

1. **HTTPS 経由のスマホアクセス確認**：
   - `npm run dev -- --host` で LAN 内の他デバイスから接続できることを確認
   - `@vitejs/plugin-basic-ssl` の自己署名証明書をスマホで信頼設定
2. **iPhone / Android で実機テスト**：
   - 自宅周辺の短いルート（500m〜1km）でテスト
   - 地図のピンチズーム・パン
   - ボトムシートのドラッグ
   - GPS 取得・現在地追従
   - 音声・振動アラート
3. **問題点の記録と修正**：
   - `docs/UI_TESTING.md`（新規作成）に発見した問題を記録
   - クリティカルな問題は即修正、軽微な問題は次フェーズに送る

#### 完了条件

- iPhone / Android の両方（少なくとも一方）で全機能が動作する
- `docs/UI_TESTING.md` にテスト結果と既知の問題が記録されている

## 開発時の共通ルール

- 実装前に必ず対象ファイルを `view` で読み、既存実装を把握すること
- バックエンド API は変更しない。フロントエンドのレイアウトと UX のみを変更する
- 既存の動作確認シナリオ（渋谷→新宿、東京駅→渋谷）でリグレッションが起きないこと
- localStorage や IndexedDB は使わない（卒研の動作環境を単純化するため）
- 大きな変更を加える前に、変更方針を要約してユーザーに確認すること
- バックエンドの起動方法は `docs/SETUP.md` を参照
- タスク完了時は `docs/CHANGELOG.md` に追記する
