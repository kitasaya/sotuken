## 研究の核となる設計原則：二層の責任分界（変更禁止・2026-06-16 確立）

本システムが扱う法規関連の事象は、必ず次の二層のいずれかに分類して扱う。
この分界は研究の中心的主張であり、安易に越境させない。

- **第1層：経路レベルの事前排除**
  - 対象：ルート自体が違反を内包し、その経路を通る限り違反が不可避なケース
  - 扱い：`custom_model` の `areas` で当該区間を回避し、違反を含まない経路を再生成
  - 例：一方通行の逆走

- **第2層：走行ガイダンス**
  - 対象：ルートは合法だが、ユーザーの走行方法次第で違反・危険が生じうるケース
  - 扱い：経路は変更せず、走行時に正しい方法を案内する
  - 例：二段階右折を要する交差点での右折手順

**この原則から導かれる禁止事項：**

- 二段階右折を **リルートのトリガーにしない**（第2層。F1で確立済み。リルート化
  するとゼロコスト性が崩れる）
- 右折の危険性（信号なし右折・小回り右折リスク等）を理由に **経路レベルで右折を
  排除しない**（第2層。例：「右折は信号ありのみ」といった実装は過剰迂回を生み、
  ゼロコスト主張を自壊させるため禁止）
- 「ユーザーが違反しうる」を理由に経路から区間を除外しない。除外してよいのは
  「経路自体が違反を含む」場合のみ（第1層の定義）

**判断根拠（論文 discussion 用）：** 第1層は経路選択そのものが違反であり別経路で
回避できる。第2層は経路は問題なく実行が問題になりうるため、経路で排除すると
過剰迂回を生み、ユーザーが正しく走行すれば本来不要なコストを上乗せする。
この線引きとその根拠の言語化自体を研究貢献と位置づける。

## 不変の制約（変更禁止）

- バックエンド API の **公開仕様** は変更しない（`POST /api/route` のリクエスト・
  レスポンス形式・フィールド名は維持）
- v3 判定ロジックの **設計意図**（edge_id ベース + 進行方向照合 + 右折 instruction
  限定 + confidence 3 段階）は維持する
- 上記「二層の責任分界」の原則を維持する
- `check_sidewalk_violation` は引き続き呼び出さない
  （`sidewalk=no` は物理的事実のタグであり法的禁止ではない。目視判断可能で
  ナビ介入不要。復活禁止）
- experiment.py / batch エンドポイントの **公開エンドポイント名** は維持
- `external_route_scorer.py` は **採点専用**。`law_checker` の判定関数を共有して
  外部ルートを採点するのみで、`rerouter.py` / 経路探索には一切干渉しない
  （自システムと Google で「同一の物差し」を保つことが研究上の公平性の根拠）
- フロントエンドの UI レイアウト（U1/U2 で構築したボトムシート構造、U3 の
  riding フルスクリーン構造）は変えない
- localStorage / IndexedDB は使わない
- 準拠率の指標として **エッジ数ベースの正規化を使わない**（O-Dペアをまたいだ
  OSMデータ粒度の差で意味を持たないため。違反エッジ数の絶対値と違反距離比率を使う）

### R2-auto：外部ルート自動採点器【優先度：高・実地検証済み】

#### 背景・動機

R2（Google Maps比較）は当初、Google ルートを目視でなぞって違反箇所を手動カウントし
`google_comparison.csv` の `google_oneway_violation_count` /
`google_two_step_violation_count` に記入する設計。15ペアの手動カウントは負担が大きく、
判定基準も人手依存でばらつく。これを **本システムと同一の判定器** で自動採点し半自動化する。

研究上の意義：Google の内部アルゴリズムが法規を考慮しているか否かは非公開で証明不能。
したがって「Google は法規を無視している（意図）」ではなく「**Google ルートに法規違反
エッジが N 件含まれていた（観測事実）**」というフレーミングで主張する。本採点器はこの
観測事実を機械的・再現可能に得るための実装。

#### 実装済みの内容（2026-06-23）

- **新規 `services/external_route_scorer.py`**
  - `score_external_route(coords)`：外部ルート座標列（`[[lng, lat], ...]`）を採点。
    `get_bulk_way_data` で最近傍 way のタグ＋geometry を取得し、ルート進行方向から
    travel_vector を構築して `law_checker` の `check_oneway_violation` /
    `check_two_step_turn` を **そのまま再利用**。判定の物差しを自システムと共有する。
  - `decode_polyline(encoded)`：Google encoded polyline を `[[lng, lat], ...]` に変換。
  - 距離等間隔サンプリング（既定 40m）・進行方向ベクトル・右折点抽出を内包。
- **`routers/experiment.py` にエンドポイント追加**
  - `POST /experiment/external-route/score`
  - リクエスト：`{ "label": str, "coords": [[lng,lat],...] | null, "polyline": str | null,
    "sample_interval_m": float }`（coords / polyline のどちらか一方。両方あれば coords 優先）
  - レスポンス：`oneway_violation_count` / `two_step_violation_count` /
    `total_violation_count` / `route_distance_m` / `sampled_points` 等。
    これを `google_comparison.csv` の対応列へ転記する。

#### 実地検証の結果（2026-06-23・詳細は `docs/CHANGELOG.md` の「タスク R2-auto」参照）

ground_truth.csv（渋谷→新宿6点）+ 偽陽性が観測された3点で検証し、2つの問題を発見・対応した。

- **oneway 偽陽性（対応済み）**：座標逆引き（Overpass 最近傍ノード検索）が、ルートが
  実際に通った way とは異なる交差小道（進行方向とほぼ直交）を誤って拾い、偽陽性3件を
  発生させていた。`external_route_scorer.py` に `_is_way_misaligned()` を追加し、
  マッチした way の軸方向と travel_vector の整合性をチェックすることで解消
  （偽陽性3→0、既存のTN/FNは不変）。**半径を狭める対応は不可**（取りこぼし＝偽陰性化を
  招くだけで構造的欠陥は直らないため）。
- **two_step_turn 偽陰性（既知の限界として記録・対応見送り）**：instruction が無い
  外部ルートでは右折地点を折れ角（45度しきい値）で近似検出するため、緩いカーブの
  右折（実測26〜31度）は検出漏れになる。**閾値は下げない**（緩いカーブの誤検出を招くため）。
  自システムは GraphHopper の instruction で正確に右折地点を取得するのに対し、Google 側
  だけ折れ角ベースの近似検出になる、という **検出条件の非対称** を論文に明記すること。

- **本採点器は判定器を共有するのみで、経路探索ロジックには一切触れない。** R2 の
  迂回コスト実測値（2026-07-07時点：14/15ペアはリルートなし・distance_diff=0.0m、
  1/15（横浜→みなとみらい）は実在する oneway=yes タグ検出により+415.8m）に
  影響を与えてはならない。
- confidence は座標ベース（way_id 完全一致ではない）。oneway は方向照合込みで
  最大 1.0 だが、v3 の edge_id 一致由来の 1.0 とは意味が異なる点に留意。
- 検証データ：`backend/data/scorer_validation_groundtruth.csv`（before/after の
  TP/FP/FN/TN・Precision/Recall）、再実行用スクリプト：
  `backend/scripts/validate_scorer_groundtruth.py`。

### R3：二層分界トレードオフの定量化【優先度：中・着手は中間発表後】

#### 背景

第2層に委ねた「右折の危険性」について、「経路レベルで排除しない」という設計判断の
妥当性を定量的に裏付ける。実走検証で得た「右折コストは無駄を生む」という直感を、
数値（迂回量）に変換することが目的。段階10-1「ルート品質評価指標」の作業と合流させる。

#### 重要な制約

- **経路探索ロジックは一切変更しない。** 生成済みルートに対する後処理スコアリング
  としてのみ実装する。R2 の迂回コスト実測値（14/15はリルートなし、1/15は
  実在する違反回避のため+415.8m）に影響を与えてはならない。

#### 実装方針（後処理スコアリング）

- 各ルートの右折地点を抽出する（`check_two_step_turn` が右折 instruction 地点の
  座標を保持しているので流用）
- 各右折を道路属性（`road_class` ローカルタグ・車線数・信号有無）で危険度づけし、
  ルート単位の「右折危険度スコア」を算出
- 「仮に右折を経路レベルで排除した場合の迂回量」を、当該右折地点を `areas` で
  ブロックした仮想ルートとの距離差として計測
- 結果を評価指標として CSV 出力（経路選択は変えない）

#### 完了条件

- 各O-Dペアについて、右折危険度スコアと「右折排除時の仮想迂回量」が CSV 出力される
- 「右折を事前排除するとこれだけの迂回コストが発生するため、走行ガイダンス（第2層）
  に委ねた」というトレードオフが数値で示せる
- `docs/CHANGELOG.md` に実装記録を追記する

#### 将来拡張の参考（今フェーズでは実装しない）

経路最適化に右折危険度を統合する場合は、`rerouter.py` の `multiply_by: 0`
（完全ブロック）を `multiply_by: 0.3` 等の部分ペナルティに可変化する方向が候補。
ただし右折地点周辺の直進エッジも巻き込む副作用があるため、半径設計の検討が必要。
理論的根拠はリンクペナルティ（De La Barra et al. 1993）・自転車経路選択
（Broach et al. 2012）。

## 論文執筆フェーズに向けて

実装はほぼ完了し、以下を論文の柱として記述していく。

- **主張：** 法規違反ルートの事前排除は技術的に実現可能。違反が存在しないルートでは
  追加コストなし（2026-07-07実測：15ペア中14ペアはリルートなし・distance_diff=0.0m）。
  実在する違反（横浜→みなとみらい：県道80号の oneway=yes）を検出した場合は迂回によって
  相応の距離増加（+415.8m）が生じるが、これは「違反を回避すれば迂回が生じる」という
  当然の挙動であり不具合ではない。旧「迂回率0/15・+0.0m」は2026-05-18時点のコードでの
  一時的なスナップショットに基づく主張であり、その後のコード変更を経て比較対象として
  不十分と判断し撤回した。現在の実測値を正データとして扱う。
- **設計貢献：** 事前排除と走行ガイダンスの二層の責任分界。何を経路で排除し、
  何をガイダンスに委ねるかの線引きとその根拠。
- **評価：** R1（検出精度 Precision/Recall/F1）、R2（Google Maps との実用コスト比較）、
  R3（二層分界のトレードオフ定量化）、ユーザーアンケート（受容性）。
- **考察材料：** Q1ロールバック（cycleway優遇と違反検出の相性）、二段階右折の
  ガイダンス化、sidewalk除外の判断、右折危険性をスコープ外とした判断。
