## 発見されたバグ：二段階右折ではリルートすべきでない

### 法的事実

道路交通法上、自転車の違反は2種類ある：

1. **一方通行違反**：その道を逆走すること自体が違法 → 別ルートで**回避すべき**
2. **二段階右折**：右折時に「直進 → 停止 → 左折」と2段階に分けて曲がる**手順を守れば合法** → 別ルートに**変える必要はない**

### 現状の実装の問題

`route_analyzer.py` で violation を検出すると、種別を問わず `rerouter` で
violation エリアを GeoJSON ポリゴンでブロックして別ルートを再探索している。

このため：

- 二段階右折検出時にも不要な迂回が発生
- 距離が無駄に増加（渋谷→新宿の例：4175m → 4439m、+264m）
- 「この道は通れない」とユーザーに誤った印象を与える
- 論文の主張（「法規準拠でも距離増加は小さい」）が弱くなる

### 本来あるべき挙動

| 違反種別                      | 検出時の動作                               |
| ----------------------------- | ------------------------------------------ |
| `oneway`（一方通行違反）      | リルートして回避（現状維持）               |
| `two_step_turn`（二段階右折） | **リルートしない**。マーカー表示と案内のみ |

### 修正の研究的意義

この修正は単なるバグ修正にとどまらず、論文の主張を強化する：

- 論文の手法章で「二段階右折は経路変更ではなく走行手順の指示として扱う設計」と
  記述できる
- 評価実験で「距離増加が小さい」という主張が、修正前データより強くなる
- 宮治先生の「成果の難易度が低い」指摘への反論材料の追加
- 論文の考察セクションに「実装当初はすべての違反種別でリルートしていたが、
  検証作業を通じて二段階右折は経路変更ではなく走行手順の指示が適切であることが
  判明し、システムを修正した」と書ける（実装の試行錯誤として明示）

## タスク一覧

### タスク F1：二段階右折のリルート除外【優先度：最高】

#### 背景

`route_analyzer.py` のリルート判定で、violation 種別を考慮せずすべての違反を
リルート対象にしている箇所を修正する。oneway 違反のみがリルート対象になるよう
にする。

#### 変更対象ファイル

- `backend/services/route_analyzer.py`（リルート判定ロジック）
- `backend/services/rerouter.py`（リルート対象 violation の絞り込み、必要に応じて）
- 場合によっては `backend/routers/route.py`（レスポンス構造）

#### 手順

1. **route_analyzer.py の現状確認**：
   - `_analyze_v3` 内で violations を集計し、rerouter を呼び出している箇所を特定
   - rerouter に渡す violation リストの作り方を確認
2. **rerouter 呼び出し前に violations をフィルタ**：
   - `reroute_violations = [v for v in violations if v["rule"] == "oneway"]`
   - oneway 違反のみを rerouter に渡す
   - two_step_turn 違反は元の violations リストには含めるが、リルート対象には
     しない
3. **レスポンスの violations は維持**：
   - フロントエンドが受け取る `violations` は全件（oneway + two_step_turn）を
     維持する。マーカー表示と案内のために必要
   - フロントエンドが「これは迂回されたか」を判断できるよう、各 violation に
     `triggered_reroute` フィールド（true/false）を追加することを検討
4. **リルートが不要だった場合の挙動**：
   - 検出された違反がすべて two_step_turn のみ → リルートを呼ばずに元ルートを
     返す（`rerouted = false`）
   - `compliant_route` には元ルートと同じものを入れる、または null にする
   - `compliant` フラグの扱いは要検討：「すべての oneway 違反を回避済み」を
     compliant とするのが自然
5. **既存テスト・動作シナリオでのリグレッション確認**：
   - 渋谷→新宿（two_step のみ検出されるケース）：リルートが発生せず、元ルート
     のまま返ること、距離が 4175m のままになること
   - 東京駅→渋谷（oneway も two_step も検出されるケース）：oneway は回避され、
     two_step は元ルート上に残ること
6. **v1 vs v3 実験の再実行**：
   - `POST /api/experiment/batch/od-pairs/compare/csv` を再実行
   - 15 O-D ペアぶんの新しい CSV を取得
   - 修正前 CSV と比較し、距離増加が縮小していることを確認
   - 結果を `docs/CHANGELOG.md` に追記

#### 完了条件

- 二段階右折のみ検出されたルートでは、`rerouted=false` が返り、元ルートのまま
  になる
- 一方通行違反が含まれるルートでは、従来通り oneway のみ回避するリルートが
  発生する
- レスポンスの `violations` には oneway と two_step_turn の両方が含まれる
- 渋谷→新宿で距離が 4175m に戻る（4439m から元に戻る）
- 15 O-D ペアでの再実行が完了し、新しい CSV が `data/experiment_v1_vs_v3_v2.csv`
  などの名前で保存される
- `docs/CHANGELOG.md` にバグ修正の経緯と効果（距離増加の縮小幅）が追記される
- `docs/ARCHITECTURE.md` の `route_analyzer.py` 説明に「two_step_turn は
  リルート対象外」と追記される

### タスク F2：フロントエンドの違反種別表示の差別化【優先度：中】

#### 背景

タスク F1 の修正後、レスポンスの violations には「迂回された違反（oneway）」と
「迂回されなかった違反（two_step_turn）」が混在する。フロントエンドでこの違いを
ユーザーに分かりやすく表示する必要がある。

このタスクは UI 改善フェーズ（U1〜U4）でまとめて実施することも可能。
F1 完了後に必要性を判断する。

#### 変更対象ファイル

- `frontend/src/components/ViolationAlert.jsx`
- `frontend/src/components/MapView.jsx`（マーカーアイコンの種別分け）

#### 手順

1. **MapView.jsx でマーカーを種別分け**：
   - oneway：赤いマーカー（「進入禁止」をイメージ）
   - two_step_turn：青いマーカー（「指示」をイメージ）
2. **ViolationAlert.jsx でメッセージを差別化**：
   - oneway：「一方通行を回避しました」
   - two_step_turn：「二段階右折してください」
3. **凡例の追加**：
   - 地図上または UI に「⚠️ 回避済み / ℹ️ 走行時注意」の凡例を表示

#### 完了条件

- 違反種別ごとに異なる色・アイコンで地図に表示される
- ユーザーが「迂回された違反」と「指示が必要な違反」を一目で区別できる

#### 備考

このタスクは UI 改善フェーズ（CLAUDE.md・UI 版）の U2 と統合できる。
F1 完了時点で進めるか、UI フェーズまで保留するかは要相談。

## 開発時の共通ルール

- 実装前に必ず対象ファイルを `view` で読み、既存実装を把握すること
- バックエンド API のレスポンス構造を変更する場合は、フロントエンドへの影響を
  確認してから実施すること

  こと
- 大きな変更を加える前に、変更方針を要約してユーザーに確認すること
