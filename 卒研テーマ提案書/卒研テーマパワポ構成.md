## スライド 2｜背景①：自転車事故・違反の現状（0:30〜2:00）

**メインメッセージ**：自転車関連事故は全交通事故の約4分の1を占め、深刻な水準にある

**掲載内容**

- 自転車保有台数：**約6,870万台**（ほぼ2人に1台）— 国土交通省
- 自転車関連事故件数：年間**約7万件前後**で横ばい

## スライド 4｜背景③：既存ナビの問題点（3:30〜5:00）

**メインメッセージ**：既存ナビは法規違反ルートを案内しうる——青切符施行後は利用者が知らずに違反者になるリスクがある

**掲載内容**

- Google Maps・Yahoo!カーナビ等は**最短・最速経路優先**で、法規への対応が不十分
- **実際に起きている問題（具体的な事例）**
  - 「自動車用一方通行（自転車を除く標識なし）」を自転車が逆走するルートを案内
  - 歩道・歩行者専用道へ誘導するルートが生成される
  - 二段階右折が必要な幹線道路で直接右折を案内
- 自転車NAVITIMEは2026年4月に一部対応（運転中アラート）したが、**ルート生成段階での法規違反排除ではない**

**スライドの工夫**

- 左：既存ナビのスクリーンショット（逆走ルートの例）／右：「何が問題か」を矢印で指摘
- 「アラートで知らせる」と「最初から違反ルートを生成しない」の違いを図で対比する
- 出典（実地検証）：サイクルガジェット記事
  - 📎 https://www.cycle-gadget.com/google-map-one-way-street-for-cyclist.html
- 出典（NAVITIMEの対応）：ナビタイムプレスリリース
  - 📎 https://corporate.navitime.co.jp/topics/pr/202603/27_5993.html

## スライド 7｜システム構成：全体アーキテクチャ（7:30〜9:00）

**メインメッセージ**：既存エンジンをOSMタグで拡張し、法規チェック→再ルーティングを実現

**掲載内容（ブロック図でビジュアル化）**

```
ユーザ入力（住所・地名）
        ↓  Nominatim API（ジオコーディング）
FastAPI バックエンド
   ├─ GraphHopper（Docker自己ホスト）  → ① 初期ルート生成
   ├─ Overpass API                      → ② OSMタグ一括取得
   └─ 法規チェックエンジン（asyncio並列）→ ③ 違反箇所を検出
        ↓ 違反あり
    GraphHopper（custom_model）         → ④ 再ルーティング
        ↓
React + Leaflet.js フロントエンド       → ⑤ 地図上に表示
```

**利用技術一覧**
| コンポーネント | 技術 | 役割 |
|---|---|---|
| ルーティングエンジン | GraphHopper（Docker） | OSM地図上の経路探索 |
| 地図データ | OpenStreetMap（関東） | 道路タグの取得元 |
| タグ取得API | Overpass API | 道路属性の一括クエリ |
| バックエンド | Python / FastAPI | 法規判定・再ルーティング制御 |
| フロントエンド | React + Leaflet.js | 地図UI・結果表示 |
| ジオコーディング | Nominatim API | 住所→座標変換 |

**スライドの工夫**

- テキストのフロー図よりも**矢印付きブロック図**として描く（PowerPointのSmartArt等が有効）
- ①〜⑤の番号で処理の流れを示す

## スライド 8｜システム構成：法規チェック詳細（9:00〜10:00）

**メインメッセージ**：OSMタグで4種類の法規を判定し、Overpassバルク化で3.4秒に高速化

**掲載内容**

| チェック項目       | 使用OSMタグ                              | 種別            | 備考                                                     |
| ------------------ | ---------------------------------------- | --------------- | -------------------------------------------------------- |
| 一方通行違反       | `oneway=yes`                             | 🔴 違反         | `oneway:bicycle=no` で自転車除外判定                     |
| 歩道なし道路の走行 | `sidewalk=no`                            | 🔴 違反（近似） | 「歩道が存在しない道路」の近似。論文で制約として明記予定 |
| 二段階右折義務     | `highway=primary/secondary` or `lanes≥3` | 🔴 違反         | 幹線道路・多車線で判定                                   |
| 自転車レーン推奨   | `cycleway=lane/track`                    | 🔵 推奨情報     | 違反ではなく優先表示                                     |

## スライド 9・10｜評価方法（案）（10:00〜12:30）

**メインメッセージ**：複数O-Dペアの定量比較実験で有用性を実証する

**掲載内容**

**実験設計**

- 関東圏内の複数の出発地・目的地ペア（O-D）で比較
- 提案システムのルート（法規準拠）vs 最短経路ルート（既存ナビ相当）

**定量実験**

- 比較指標
  - 法規違反箇所数の差分
  - 距離・走行時間のオーバーヘッド（倍率）
  - 準拠率（全チェックエッジのうち違反なし判定の割合）← 定義は検討中
- ベンチマーク
  - 縄野・間邊（2020）：96.9%が最短経路の1.3倍以内
  - Winters et al.（2010）：実利用者は安全ルートのために平均360m迂回を許容

**定性評価**

- 法規準拠ルートへの受容性（使いたいと思うか）
- 距離増加の許容度
- 違反箇所の表示UIのわかりやすさ

**スライドの工夫**

- 「比較実験のイメージ図」（関東地図上に複数O-Dペアを散布した図）があると具体的に見える
- 「準拠率の定義（検討中）」は口頭で「宮地先生にご意見いただきたい」と伝え、質疑を誘導する
